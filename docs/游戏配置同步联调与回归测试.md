# 游戏配置同步联调与回归测试

## 1. 测试目标

本文用于验证 OTA 应用新增的游戏配置同步能力是否满足以下目标：

1. `command/config` 能进入游戏配置状态机。
2. OTA 在 pending 可靠保存后才上报 `command-result ack`。
3. OTA 能通过签名权限广播把配置下发给 Unity。
4. Unity 返回 success 后，OTA 能保存 applied 配置并上报 `command-result success`。
5. OTA 能上报 `report/game-config` 当前实际配置。
6. MQTT 离线时不会丢失 `command-result` 和 `report/game-config`。
7. OTA 重启、MQTT 重连、Unity 数据被清理后能够恢复配置。
8. 原有游戏 APK 升级、球盘 BIN 升级、OTA 自升级流程不受影响。

## 2. 测试前置条件

### 2.1 构建环境

在 Windows 项目根目录执行：

```bat
gradlew.bat clean testDebugUnitTest assembleDebug
```

```bash
./gradlew clean testDebugUnitTest assembleDebug
```

构建必须满足：

```text

所有 gameconfig 单元测试通过
生成 app/build/outputs/apk/debug/*.apk
```

### 2.2 设备环境

设备必须满足：

```text
OTA 包名：com.zeda.ota
Unity 包名：com.zeda
OTA 与 Unity 使用相同签名
设备已激活
设备可连接 MQTT Broker
MQTT 固定订阅 QoS = 1
MQTT 固定发布 QoS = 1
MQTT Retain = false
```

### 2.3 Unity 侧最低实现

Unity 至少实现：

```text
1. 声明 com.zeda.permission.GAME_CONFIG
2. 注册 com.zeda.ota.action.GAME_CONFIG_REQUEST 接收器
3. 处理 apply_game_config
4. 启动后发送 game_config_ready
5. 保存 Application.persistentDataPath/game_config/game_config.json
6. 返回 com.zeda.action.GAME_CONFIG_EVENT
7. 通过 8 组哈希测试向量
```

## 6. 异常配置测试

### 6.1 configHash 错误

将 EASY 指令中的 `configHash` 改为任意错误 64 位小写十六进制。

预期：

```text
OTA 不广播给 Unity
OTA 上报 command-result failed
resultCode = CONFIG_HASH_MISMATCH
pending_record 不应保存
applied_record 不应更新
```

### 6.2 schemaVersion 不支持

将 `schemaVersion` 改为 `2`。

预期：

```text
OTA 不广播给 Unity
OTA 上报 command-result failed
resultCode = SCHEMA_UNSUPPORTED
```

### 6.3 低版本

设备已生效 `configVersion = 13` 后，再下发 `configVersion = 12`。

预期：

```text
OTA 不广播给 Unity
OTA 上报 command-result failed
resultCode = VERSION_STALE
```

### 6.4 相同版本不同哈希

设备已生效 `configVersion = 13` 后，再下发 `configVersion = 13` 但 `configHash` 不同。

预期：

```text
OTA 不广播给 Unity
OTA 上报 command-result failed
resultCode = VERSION_CONFLICT
```

### 6.5 Unity 返回 failed

Unity 返回：

```json
{
  "protocolVersion": 1,
  "operation": "apply_game_config",
  "requestId": "本次 OTA 下发的 requestId",
  "messageId": "5e20c7f6e6354a29acb5d44d01736b8f",
  "status": "failed",
  "resultCode": "APPLY_FAILED",
  "resultMessage": "游戏难度模块尚未初始化",
  "timestamp": 1784200005000,
  "data": {}
}
```

预期：

```text
OTA 清除 pending_record
OTA 不更新 applied_record
OTA 上报 command-result failed
OTA 不上报新的 report/game-config
```

## 7. 幂等测试

### 7.1 重复 messageId

下发同一个 `messageId` 两次。

预期：

```text
如果第一次已完成：OTA 直接重放最近结果，不重复广播给 Unity
如果第一次仍 pending：OTA 重新 ack 并重新派发 pending
```

### 7.2 相同 configVersion + 相同 configHash

使用不同 `messageId`，但 `configVersion` 和 `configHash` 与已生效配置一致。

预期：

```text
OTA 可直接 success
不得产生 VERSION_CONFLICT
```

## 8. MQTT 离线与补报测试

### 8.1 command-result outbox

步骤：

```text
1. 断开 MQTT 网络
2. 下发配置或让 Unity 返回结果时 MQTT 发布失败
3. 检查 OTA 本地保留 command_result_outbox
4. 恢复 MQTT
5. OTA 自动补发 command-result
6. 发布成功后清除 command_result_outbox
```

### 8.2 game-config report outbox

步骤：

```text
1. Unity 返回 success 后断开 MQTT
2. OTA 保存 game_config_report_outbox
3. 恢复 MQTT
4. OTA 自动补发 report/game-config
5. 发布成功后清除 game_config_report_outbox
```

## 9. OTA 重启恢复测试

### 9.1 pending 恢复

步骤：

```text
1. 下发配置
2. OTA 已保存 pending_record 后，杀掉 OTA 进程或重启设备
3. OTA 启动并连接 MQTT
4. OTA 重新广播 pending 配置给 Unity
```

预期：

```text
Unity 收到 apply_game_config
Unity success 后 OTA 清除 pending_record
OTA 保存 applied_record
OTA 上报 command-result success
OTA 上报 report/game-config
```

### 9.2 applied 恢复

步骤：

```text
1. 先成功应用一个配置
2. 确认 OTA 存在 applied_record
3. 清除 Unity 数据或升级游戏 APK 后由现有流程 pm clear com.zeda
4. Unity 启动后发送 game_config_ready
5. OTA 重新下发 applied 配置给 Unity
```

预期：

```text
Unity 重新保存 game_config.json
OTA 不创建新的后端配置版本
OTA 可补发 report/game-config
```

## 10. Unity 广播安全测试

### 10.1 签名不一致

使用不同签名的 Unity APK。

预期：

```text
广播不可互通
Unity 不应收到 OTA 请求，或 OTA 不应接收 Unity 事件
```

### 10.2 非目标包广播

Unity 返回事件时不设置：

```java
intent.setPackage("com.zeda.ota")
```

预期：

```text
视 Android 版本和系统策略可能被拦截或无法稳定到达
正式版本必须设置目标包
```

### 11.1 游戏 APK 升级


## 13. ADB 检查命令

查看 OTA 日志：

```bat
adb logcat -s OTA_TEST
```

查看 Unity 数据目录是否存在配置文件：

```bat
adb shell "run-as com.zeda ls files/game_config"
```

如果 `run-as` 不可用，可在 root 设备上检查：

```bat
adb shell "su 0 ls -l /data/data/com.zeda/files/game_config"
```

查看 OTA 进程：

```bat
adb shell "pidof com.zeda.ota"
```

查看 Unity 进程：

```bat
adb shell "pidof com.zeda"
```

## 14. 当前静态检查备注

当前 MQTT 相关源码文件位于：

```text
app/src/main/java/com/zeda/ota/mqtt/
```

但 Java 包名仍为：

```text
package com.zeda.ota;
```

Java 编译不强制要求文件夹路径与 package 完全一致，但这会增加维护风险。若后续本地 Gradle 构建或 IDE 索引出现异常，建议单独做一次路径整理，把文件移动到：

```text
app/src/main/java/com/zeda/ota/
```

本次第 10 步不移动文件，避免对现有工程结构产生额外影响。
